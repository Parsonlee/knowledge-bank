# 概念：LLM 应用评估体系

## 定义

LLM 应用评估体系是指在大模型应用开发全生命周期中，对 Prompt/RAG/Agent 输出质量进行系统性、可重复评估的方法论与工具集合。

## 核心原则

### 1. 断言式单元测试

基于生产输入/输出样本，至少设置 3 个评测指标：
- 少于 3 个指标表明任务未充分定义或过于开放
- 工作流任何更改（提示词/RAG/模型）都应触发测试

### 2. LLM 裁判（LLM-as-Judge）

用最强模型对其他模型输出打分——**定性比较有用，具体幅度无参考价值**。

最佳实践：
- 成对比较优于量表打分（选择更好的一个，结果更稳定）
- 每对交换顺序各做一次（消除位置偏差）
- 允许宣布平局
- 搭配 CoT（解释决定后再给偏好），可让小模型达到大模型效果
- 注意 LLM 偏向较长回答

### 3. "实习生测试"

> "如果将提示词作为任务交给相关专业的普通大学生，他们能成功吗？"

判断逻辑：
- 大学生做不到 → 需要提供更丰富上下文
- 无法通过改进上下文解决 → 任务对当代 LLM 太难
- 大学生能做到但很慢 → 尝试降低复杂性/拆解任务
- 大学生能快速完成但 LLM 不行 → 深入研究失败模式

### 4. 古德哈特定律警惕

"当一项指标成为目标时，它就不再是一项好指标。"

典型案例：大海捞针测试被快速刷榜后失效。更好的指标是复杂实际任务（如会议记录总结特定要素），超越死记硬背，考察解析/识别/归纳综合能力。

### 5. 指标过强化的副作用

- 强调事实一致性 → 摘要变得不具体（为避免与事实不符）
- 强调写作风格 → 更多花哨话术（可能与事实不符）

## 幻觉处理

幻觉率降至 2% 以下极难，即使在摘要等简单任务中。双层应对：
- **生成上游**：提示工程（CoT 让模型先推理）
- **生成下游**：事实不一致护栏（检测/过滤/重生成）

同时需持续记录输入输出以供调试监控。
