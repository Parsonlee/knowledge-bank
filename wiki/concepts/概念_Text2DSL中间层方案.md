# 概念：Text2DSL 中间层方案

## 定义

将自然语言转换为 BI 平台特定 DSL（Domain Specific Language，描述数据分析条件的协议语言）时，引入 SQL 作为中间层，分两步完成转换：NL → SQL → DSL → 前端指令。

## 问题背景

DSL 是 BI 平台特定的查询协议，包含指标/维度/过滤条件/时间范围/排序等结构化定义。直接从自然语言生成 DSL 困难，因为 DSL 语法与自然语言差距大，且各平台 DSL 格式不同。

## 方案设计

```
自然语言
   ↓ Text2SQL（已有成熟方案）
   SQL（中间层）
   ↓ SQL解析（group by / having / where / 子查询）
   DSL（指标/维度/过滤/时间/排序）
   ↓ 平台适配
   前端指令
```

**SQL 解析过程**：
- 单层 SQL：提取 group by / having / where，生成维度/指标/过滤条件
- 多层 SQL（含子查询）：各层字段通过别名连接，展开后生成完整查询结构
- 复杂函数（平台自定义/窗口函数）：生成虚拟字段，用 SQL 语句描述指标

## 关键洞察：无损转换的重要性

> 每层转换都有信息损耗。若每层转换率为 0.9，三层后最终结果为 0.9³ ≈ 0.73，显著下降。

因此每层转换需最大化保留信息，中间层不是越多越好。

## 引申原则

**当直接生成目标结果困难时，引入中间层是通用工程手段**：
- 直接 NL→DSL 困难 → 引入 SQL 中间层
- 直接 NL→代码 困难 → 引入伪代码/逻辑描述中间层
- 关键约束：中间层需设计为"无损"或"损耗最小"
