# 概念：上下文隔离两种模式

## 背景

多Agent系统中，子Agent间如何共享信息是核心难题。借鉴 Go 语言哲学：
"Do not communicate by sharing memory; instead, share memory by communicating."

## 模式一：通过通信（By Communicating）

**工作流**：主Agent封装清晰指令 → 子Agent以干净上下文独立完成 → 返回结果

**适用场景**：指令简短清晰、主Agent只关心最终产出、不关心实现过程

**典型案例**：代码库搜索；Claude Code 的 task 工具

**优点**：简单、隔离性好、上下文开销小

## 模式二：通过共享上下文（By Sharing Context）

**工作流**：子Agent能看到主Agent完整历史，但拥有自己独特的系统提示和行动空间

**适用场景**：产出质量依赖大量中间历史（如深度研究报告）

**Manus实现**：共享VM沙箱（文件路径传递）+ Schema作为合约（submit_result + 约束解码）

**成本**：预填充成本高；每个子Agent系统提示不同导致KV缓存失效

## 选择原则

- 任务可清晰分解且产出独立 → 通信模式
- 产出强依赖完整历史背景 → 共享上下文模式
