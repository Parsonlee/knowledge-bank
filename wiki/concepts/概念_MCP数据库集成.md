## 定义

MCP 数据库集成是指通过 MCP 协议将数据库（关系型、文档型、图数据库等）暴露给 AI Agent，实现类似 Text-to-SQL 的自然语言结构化检索，作为传统 RAG 在结构化数据场景的替代或补充方案。

## 传统 RAG 的四大局限（结构化数据场景）

1. **检索精度不足**：向量相似度匹配在精确查询上存在低精确率/低召回率问题（如"身高 180-190cm 的女生"）
2. **生成内容不完整**：文档切片的局部性导致无法看到完整记录
3. **缺乏大局观**：无法判断文档/记录间的关系（如跨表关联查询）
4. **多轮检索能力弱**：复杂查询需要多轮检索时，RAG 的误差会累积

## MCP+数据库的优势

- **精确查询**：LLM 生成 SQL/查询语句，数据库引擎执行，结果精确
- **完整数据**：直接查询原始记录，无切片损失
- **关联查询**：支持 JOIN、聚合等结构化操作，RAG 无法实现
- **动态扩展**：新增数据表无需重新建索引，即时可用

## 实战：VS Code + Cline + mcp-mongo-server

### 配置方式（Stdio 模式）

```json
// .vscode/settings.json 或 Cline MCP 配置
{
  "mcpServers": {
    "mongodb": {
      "command": "npx",
      "args": ["-y", "mcp-mongo-server", "mongodb://localhost:27017/mydb"]
    }
  }
}
```

### 优化技巧

在全局提示词中**预先提供表结构说明**，避免模型猜测字段：

```
数据库包含以下集合：
- students: {name, gender, height, age, grade}
- ...
```

效果：提升查询准确性，减少模型试探性查询带来的 Token 消耗。

## 与 RAG 的实测对比

| 查询类型 | RAG 效果 | MCP+数据库 |
|---------|---------|-----------|
| "身高 180-190cm 的女生" | 差（向量相似度无法处理范围查询）| 好（直接 SQL） |
| "姓李的男同学" | 差（部分匹配不稳定）| 好（精确字符串匹配）|
| "按身高排序的前10名" | 不支持 | 好（ORDER BY LIMIT）|
| "自然语言问答类问题" | 好（语义匹配）| 一般（依赖 Schema 质量）|

**结论**：结构化数据检索场景，MCP+数据库明显优于传统 RAG。

## 当前局限

1. **Token 消耗高**：每次真实执行 SQL 查询大量数据会消耗大量 Token，极端情况卡死
2. **系统提示词开销**：MCP 工具定义本身大幅增加 Token 消耗
3. **实时执行风险**：每次查询都真实操作数据库，误操作风险需要权限控制

## 适用场景

- **适合**：智能客服（查订单/查用户）、仓储管理、信息管理、HR 系统、CRM
- **不适合**：非结构化内容检索（文章、知识库问答）— 仍需 RAG

## 与 MCP Resource 原语的关系

- **MCP Tool**：执行查询（适合动态、实时的数据操作）
- **MCP Resource**：暴露数据库 Schema（适合让 AI 了解数据库结构，不执行操作）

最佳实践：用 Resource 暴露 Schema，用 Tool 执行查询，两者配合使用。
