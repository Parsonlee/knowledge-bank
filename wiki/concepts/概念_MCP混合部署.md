# 概念_MCP混合部署

在 MCP 工作流中，用不同规模的模型承担不同步骤，平衡 token 成本与输出质量。

## 动机

MCP 工具选择步骤需要将**所有工具的 description 文本**传给模型，工具越多 token 消耗越大。而结果生成步骤对模型能力要求高，需要大模型保证稳定性。

## 典型分工

| 步骤 | 任务 | 推荐模型 | 原因 |
|------|------|--------|------|
| 工具选择 | 读取工具描述，输出结构化 JSON 调用请求 | 本地小模型（如 Qwen3-0.6b） | 节约 token，本地部署无 API 费用 |
| 结果生成 | 读取工具执行结果，生成自然语言回答 | 大模型（如 Qwen3-235b-a22b） | 保证输出质量和稳定性 |

## 实现要点

- 工具选择模型通过 Ollama 等框架本地部署，使用 OpenAI 兼容接口
- 两个模型分别初始化为 client1（本地）和 client2（云端 API）
- 工具 docstring 质量决定小模型选择准确率

## token 参考

单个简单工具描述约消耗 182 tokens；多个描述复杂的工具将大幅放大成本，小模型选工具可显著节省。

## 前提条件

MCP 工具选择本质是 prompt engineering：任何模型只要能理解工具描述文本都可参与，与模型提供商无关。
