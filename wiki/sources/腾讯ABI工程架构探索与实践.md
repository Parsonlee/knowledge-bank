# 腾讯在 ABI 工程领域的探索与实践

## 来源信息

- 作者：霍琦（腾讯 PCG 大数据 ABI 工程架构负责人）
- 来源：DataFun 社区
- 标签：`AI-Agent/AI-BI`

## 核心要点

### 1. 背景与挑战

腾讯内部 ABI 工程侧面临四大问题：
- **架构不清晰**：快速迭代导致耦合，AI 能力难以跨平台复用
- **单 Agent 设计复杂**：集成太多业务属性，扩展性差
- **LLM 实现难度大**：上下文管理、长文本记忆、幻觉问题
- **数据分析整合难**：传统能力难以融入智能化场景

### 2. OlaChat 技术框架

**全链路数据分析能力**：
- 意图理解：多轮对话 / Query 改写 / 意图分类与识别
- 数据分析：找数据（表查找/表关系）/ 数据开发（Text2SQL/SQL纠错/改写解读）/ 端到端分析（指标分析/Text2DSL）
- 数据解读：描述性统计 / 异常检测 / 归因分析 / 趋势预测
- 知识与推荐：数据知识助手 / 指标维度推荐 / 表字段推荐 / Query 推荐

**产品演进路径**：大屏展示 → Copilot 模式 → AI 原生嵌入（将 AI 组件嵌入图表分析/拖拽分析等现有产品，不改变用户习惯）

### 3. 架构核心设计

**五层架构**：
1. 前端组件层：Copilot / 魔法棒 / 输入模块 / 大屏分析
2. 架构编排层：原子 Agent 编排 + 流式协议分发
3. 原子 Agent 层：意图识别/Text2SQL/数据解读等拆成原子操作
4. 数据服务层：SQL 类服务 + 指标 HeadlessBI + 图表服务 + RAG / 高频 query / 上下文理解
5. 大模型层：腾讯混元（通用）+ 域微调模型

### 4. 编排层关键设计

**单 Agent 多 Step**（以 SQL 纠错为例）：
- 步骤1：告知错误原因（权限/字段错误等）
- 步骤2：给出修正建议
- 步骤3：提供正确 SQL
- 通过 DAG 图编排确保返回顺序，支持并行/串行配置

**多 Agent 协作**（以指标分析为例）：
- 原子 Agent 分工：找表 → 找指标 → 生成 SQL → 查询出图
- 高频查询预定义（命中则直接返回，无需重新生成）
- 指标统一语义层 + 数据源可用性检查 + 降级到 Text2SQL

**统一协议**：[重点]
- 所有 handler 遵循统一输入输出协议，agent 之间在编排框架中自动流转
- 无需硬编码，简化自动化编排
- 支持"伪装为 Agent 的人工干预"：人工输入遵循与 Agent 相同的协议，保持流程连贯性

### 5. Agent 关键能力提升

**对话上下文优化**：
- 精简意图候选集（减少可能的意图类别）
- 场景导向意图识别（缩小识别范围）
- 从历史对话提取关键信息，过滤无效数据，减少幻觉和 Token 压力

**Text2SQL 准确性提升**：
- 高频查询指标候选集（验证过的高质量指标预查询）
- 常用表字段候选集（优先推荐用户常用表）
- 数据预览/抽样确认字段类型
- 生成后 SQL 可执行性校验 + 迭代修正

**Text2DSL（引导补全）**：[重点]
- 问题：自然语言→DSL（拖拽分析协议）直接转换困难
- 方案：引入 SQL 作为中间层（NL→SQL→DSL→前端指令）
- 关键洞察：**无损转换非常重要**，每层转换都有损耗（如0.9的转化率），多层叠加后最终效果显著下降
- SQL 解析出维度/指标/过滤条件/时间/排序字段，单层/多层分别处理，复杂函数用虚拟字段

### 6. 流式协议与监控

- **SSE**：一次请求多次响应（找表→生成SQL→SQL解读流式返回）
- **WebSocket**：双向通信（SQL 补全实时反馈）
- 监控：每次请求的耗时监控 + 详细执行轨迹记录

### 7. 未来规划

- SQL IDE 原生体验（深耕 SQL 场景）
- 数仓智能化（主题理解 / 任务链路推荐 / 异常代码识别）
- 自动化标注提升效率
- 画布形态的引导式数据分析（多路并行推荐）

## 关键引文

> "当从自然语言直接生成目标结果变得困难或效果较差时，引入中间层（如 SQL）可以使得系统更加灵活和高效。" [高亮]

> "要确保每一层的转换过程尽量无损，以避免在多次转换中丢失信息。如果每一层都产生一定的损耗（如 0.9 的转化率），最终的结果将显著降低。" [重点]

> "人工输入被设计成类似 agent 的角色，遵循与其他 agent 相同的输入输出协议，确保整体流程的连贯性。" [重点]
