# 用系统架构思维，告别"意大利面条式"系统提示词

## 来源信息

- 作者：灵变（阿里云开发者）
- 原文：阿里云开发者公众号
- 标签：`AI-Agent/prompt-engineering`

## 核心要点

### 问题：扁平规则堆砌的三大困境

以 Dia 浏览器系统提示词为案例，指出"规则清单式"设计的根本缺陷：

1. **规则打架，行为摇摆不定**：多条互斥条件（如 simple answer 同时有"尽量包含"/"列表内容不包含"/"休闲对话不包含"）导致模型在边界情况下"猜"而不是遵循确定逻辑。
2. **越改越乱，最终没人敢动**：所有逻辑耦合在一起，任何改动可能引发雪崩效应，像没有变量/模块的老旧代码库。
3. **响应像"开盲盒"，核心价值被稀释**：有限的注意力被琐碎规则消耗，核心价值无法稳定传递。

### 解决方案：系统架构思维四层模型

将提示词设计类比软件架构，构建四层结构：

#### 第一层：核心定义（我是谁？我为何存在？）

- **角色建模**：身份 + 人格 + 立场，是多规则冲突时的最高仲裁者
- **目标定义**：功能性目标 + 价值性目标 + 质量标准/红线

#### 第二层：交互接口（我如何与世界互动？）

- **输入规范**：用标签（`<user_query>`、`<pdf_content>`、`<history>`）包裹不同来源输入，明确优先级，设安全过滤
- **输出规格**：独立于内部逻辑的"表示层"，定义响应结构、格式化规则、禁用项清单

#### 第三层：内部处理（我如何思考与行动？）

- **能力拆解**：将能力拆成高内聚低耦合的"技能模块"（如 `[SimpleAnswer_Handler]`、`[Media_Inserter]`），每模块只负责一件事
- **流程设计**：定义调用顺序和决策逻辑（"在策略规划阶段，如果查询类型是'休闲对话'，则禁用 SimpleAnswer_Handler 模块"）

#### 第四层：全局约束（我不能做什么？）

- **硬性规则（Hard Rules）**：最高执行优先级，涉及安全/伦理/法律红线
- **求助机制（Help Mechanism）**：无法处理时的固定行为模式

### 提示词设计画布模板

提供"即插即用"的 Canvas 模板，强制从系统层面思考，在写具体规则前先确立清晰架构。核心结构：

```
第一层：核心定义
  - 角色建模（身份/人格/立场）
  - 目标定义（功能/价值/红线）
第二层：交互接口
  - 输入规范（输入源/优先级/安全过滤）
  - 输出规格（响应结构/格式规则/禁用项）
第三层：内部处理
  - 能力拆解（技能模块）
  - 流程设计（标准步骤/决策逻辑）
第四层：全局约束
  - 硬性规则
  - 求助机制
```

### 核心洞察

> "一个清晰的角色设定，是解决'规则打架'问题的最高仲裁者。当多条规则冲突时，AI 可以回归其核心'人格'来做出最符合其身份的决策。" [重点]

> "将'思考什么'与'如何呈现'分离，是解决'越改越乱'问题的核心手段。" [重点]

> "通过能力拆解，使得系统维护变得前所未有的清晰和安全——每个模块只负责一件事。" [高亮]
