## 来源信息

- 标题：浅谈上下文工程｜从 Claude Code、Manus 和 Kiro 看提示工程到上下文工程的转变
- 来源：阿里云开发者（作者：单从）

## 核心要点

### 上下文工程定义
构建动态系统，以合适的格式为LLM提供正确的信息和工具，让LLM合理完成任务。核心区别：上下文≠单一prompt，是模型生成响应前看到的**一切信息**。

**七个核心组成部分**：
1. 指令/系统提示词
2. 用户提示词
3. 对话历史（短期记忆）
4. 长期记忆（跨对话持久知识库）
5. 检索信息（RAG）
6. 可用工具
7. 结构化输出

**提示工程 vs 上下文工程**：
- 提示工程：词句技巧、只限任务描述表达、"贴便签"
- 上下文工程：提供全面上下文、包含文档/示例/规则/模式/验证、"写剧本"

### Context-Rot 问题（上下文腐化）
表现：幻觉持续带偏、信息冲突不可预测、关键信息被稀释、行动瘫痪
系统化解决方法论（LangChain 四类）：
1. **Offload（卸载）**：引用代替内容，外化到文件系统/数据库
2. **Retrieve（检索）**：RAG / 文本检索（llms.txt + grep/find，Claude Code的方式）
3. **Reduce（压缩）**：摘要、rerank、语义总结（注意引入幻觉的风险）
4. **Isolate（隔离）**：分而治之，SubAgent独立上下文

### Claude Code 的上下文工程实践

**三层记忆架构**：
- 短期记忆：当前对话
- 中期记忆：智能压缩（92%阈值触发，8段式结构化保存）
- 长期记忆：CLAUDE.md 项目知识库，跨会话恢复背景和偏好

**实时 Steering 机制**：
- 异步消息队列 + 主循环双引擎
- 支持实时中断和恢复，流式输出持续反馈

**分层多 Agent 协作**：
- 主Agent协调 + SubAgent执行
- 调度器控制最多10工具并发，独立执行环境避免上下文污染

**动态上下文注入**：
- 自动识别文件引用，推荐基于依赖关系分析
- 容量控制：最大20文件 8K Token

### Manus 的优化实践（六点提炼）

1. **KV缓存优化**：输入/输出比100:1，缓存vs非缓存10倍成本差，前缀稳定/追加式/不加时间戳
2. **工具遮蔽非移除**：状态机管理可用性，logits掩码，避免KV缓存失效
3. **文件系统作为终极上下文**：无限容量、天然持久、外化记忆
4. **通过复述操控注意力**：不断重写todo列表，避免"丢失在中间"
5. **保留错误内容**：失败路径是负面样本，模型从中学习
6. **避免少样本示例陷阱**：引入结构化变化，打破固定模式

### Spec-Driven Development（规范驱动开发）
- Vibe Coding局限：高质量prompt不现实、缺乏文档约束、技术债务
- Spec-Driven：`Prompt → Requirements → Design → Tasks → Code`
- Kiro 实现：requirement.md（EARS格式）+ design.md（架构/接口/数据库）+ tasks.md（TODO列表）

### 未来演进：提示词工程 → 上下文工程 → 环境工程
| 阶段 | 关注点 | 特点 |
|------|--------|------|
| 提示词工程 | 单条高质量Prompt | 静态、依赖人类编写 |
| 上下文工程 | 动态收集组织多维度上下文 | 以模型为中心，关注"模型输入" |
| 环境工程 | 构建持续演化、可感知可交互的智能环境 | AI主动感知塑造环境 |
