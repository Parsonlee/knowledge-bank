# 腾讯基于 LLM 的智能数据分析平台 OlaChat 的落地实践

## 来源信息

- 作者：谭云志（腾讯 PCG 大数据平台部，ABI 平台算法负责人）
- 来源：DataFun 社区
- 标签：`AI-Agent/AI-BI`

## 核心要点

### 1. 从传统 BI 到智能 BI 的演进

三阶段：
- **传统 BI**：自上而下，开发周期长（1周+），沟通成本高
- **移动互联网敏捷分析**：拖拽自助，但存在学习门槛（环比等复杂操作）
- **智能 BI（LLM 时代）**：自然语言交互，降低技术门槛

LLM 对数据分析的四点改进：语言能力 / 工具使用 / 逻辑推理 / 上下文学习（In-context Learning）

### 2. OlaChat 平台架构

依托腾讯 PCG 大数据平台"Ola"和"灯塔"的元数据与行为日志，构建四层架构：

- **多任务对话系统**：上下文理解 + 意图识别 + NLU/DST/DPL/NLG 四模块
- **任务编排引擎**：DAG 图编排，理解意图后规划执行步骤
- **AI+BI 工具箱**：Query 改写 / Text2SQL / 指标分析等原子工具
- **公共服务**：统一 LLM 调度（混元 + 微调模型） + 知识检索增强 + 标注系统

### 3. 元数据检索增强（核心难点）

**为什么通用 RAG 不适用于元数据**：
- 元数据按层次结构组织（表→字段→维度），无自然语言中的前后顺序规律
- 特定修饰词关键（"有效播放次数"≠"付费播放次数"）
- 字段名可能极短，传统 embedding 检索难以精准捕获

**两种解决方案**：

1. **FlattenedRAG**：将结构化元数据打平为非结构化自然语言（"腾讯视频男性活跃用户数有多少？"），然后进行 embedding 检索 + 分类模型排序。优点：直接套用标准 RAG；缺点：维度组合多时面临组合爆炸。 [重点]

2. **StructuredRAG**：优先检索核心元素（如"活跃用户数"指标），再围绕核心进行二次检索（匹配"男性"相关维度），最终排序返回。优点：长尾多维度问题更有效。 [重点]

### 4. Text2SQL 实践

**真实业务问题**：
- 数据隐私限制（大模型协议问题），必须自训练模型
- 开源数据集偏英文、偏简单（单表<10字段），真实场景可达百字段
- 用户问法个性化，模型抗噪声能力不足（BIRD~70%）
- 冷启动缺少高质量 query→SQL 数据

**解决方案：微调大模型 + Agent**：
- 数据生成流程：内部数据脱敏 → 随机抽样构造 prompt → 数据增强循环（准确性校验 + 多样性去重 + 难度均衡：easy/medium/hard/extra-hard）
- 效果：GPT-4 准确率 32%，自训 70B 微调模型可达 52%
- Agent 辅助流程：字段精选（减冗余）→ 问题规范化 + fewshot 检索 → 后验纠错（LLM 审核生成的 SQL）

### 5. Text2SQL 之外

系统支持：改写 / 纠错 / 优化 / 解读 / 问答 / 补齐等多种能力，通过不同 Agent 组合实现。

取数模型用 8B，NL2SQL 用 70B 微调模型。

## 关键引文

> "元数据是按特定的结构、层次组织的。指标、字段、维度的组合没有固定的前后顺序或自然语言中的常见规律……通用 RAG 在解决元数据检索增强任务时存在一定的不适用性。" [高亮]

> "在我们内部，已经建立了一套数据生成的逻辑流程……特别重点生成困难类型的数据，以弥补开源数据集的不足。" [重点]
