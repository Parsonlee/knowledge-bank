# RAG挑战赛冠军方案

## 摘要

企业年报智能问答比赛冠军方案解析（源码：github.com/IlyaRice/RAG-Challenge-2）。任务：2.5 小时内解析 100 份最长达 1000 页的 PDF 年报构建数据库，回答 100 个随机问题，每个答案需注明引用页码。冠军方案在基础 RAG 之上加入多路由器、LLM 重排序等模块。

## 比赛任务

- 解析 100 份随机公司年报（PDF，最长 1000 页），2.5 小时内建库
- 回答 100 个模板生成的问题，答案类型为：是/否、公司名、职位/产品名、数值指标
- 每个答案必须注明引用页码作为证据，防止幻觉

## 1. 解析（Parsing）

- PDF 解析难点：表格结构、多栏文本、图表、字体编码问题（甚至遇到凯撒密码加密文档）
- 尝试约 20 余种解析器，结论：**没有任何解析器能完美还原 PDF**
- 最佳解析器：**Docling**（IBM 出品，比赛协办方之一）
- 重写 Docling 源码方法，导出含元数据的 JSON，再构建 Markdown（表格转 HTML 还原度近完美）
- GPU 加速：租用 4090 虚拟机（每小时 70 美分），100 份文档解析约 40 分钟

## 文本清理与表格序列化

- 用十几个正则表达式清理解析噪点
- **表格序列化**：将大表格转化为一系列小的上下文独立字符串（行式序列化/属性-值对匹配）
  - HTML 格式输入 LLM 优于 Markdown（更好理解合并单元格等复杂结构）
  - GPT-4o-mini 也能近乎无损序列化
  - 序列化结构：subject_core_entity + information_block
  - **注意：最终冠军方案并未采用序列化**（实测略微降低有效性，因 Docling 已足够好，额外文本降低信噪比）

## 2. 内容摄取（Ingestion）

### 术语统一
- 文档（document）：日常含义（一份报告）
- 块（chunk）：存储在数据库中被切分的文本片段

### 分块（Chunking）
- 每页文本切分为 300 token（约 15 句）的块
- 使用带自定义 Markdown 词典的递归拆分器
- 50 token 重叠避免信息被切断
- 每个块存储 ID 及父页面编号（metadata）
- **切分精确度对检索系统影响极小**

## 向量化（Vectorization）

- **100 个独立数据库（1 库 = 1 文档）**，因目标信息总是限制在单个文档内
- 使用 **FAISS**，IndexFlatIP 方法
  - Flat 索引：暴力搜索，精度高（适合小数据集）
  - 大于十万元素可考虑 IVFFlat 或 HNSW（ANN 速度快但牺牲精度）
  - IP（内积）= 余弦相似度，通常优于 L2
- 嵌入模型：**text-embedding-3-large**

## 3. 检索（Retrieval）

探索过的方法：
- **混合搜索（vDB + BM25）**：基础实现常降低质量，未采用
- **交叉编码器重排序（Cross-encoder）**：更精确但慢，因 API 稀缺最后放弃（推荐 Jina Reranker）
- **[重点] LLM 重排序**：最终选择
  - 让 LLM 给文本相关性打分（0~1，精确到 0.1）
  - 结构化输出：reasoning + relevance_score 字段
  - 一个请求同时发送三页提高速度
  - 加权平均：vector_weight=0.3，llm_weight=0.7
  - GPT-4o-mini 重排序每问题不到一美分

### 父页面检索（Parent Page Retrieval）
- 找到 top-N 块后，仅作为指向整页的指针，将完整页面加入上下文

### 整合后的检索器
1. 查询向量化
2. 找 top-30 相关块
3. 通过 metadata 提取对应页面（去重）
4. LLM 重排序处理页面
5. 调整页面相关性得分
6. 返回 top-10 页，加页码合并成字符串

## 4-5. 增强与生成（Augmentation & Generation）

### 多查询路由

- **路由到数据库**：从查询提取公司名（re.search 即可，无需 LLM）匹配对应数据库，搜索空间缩小 100 倍
- **路由到提示词**：根据答案类型（int/float、bool、str、list[str]）选择对应指令集（if-else），减少每查询规则数
- **复合查询路由**：比较多公司的问题先拆解为子问题分别检索，再合并回答

### CoT + 结构化输出

- **思维链（CoT）**：让模型推理后再答，明确引导推理步骤、解释目标、提供示例
- **结构化输出（SO）**：Pydantic/JSON schema 强制格式，字段描述也作为 Prompt
- **CoT SO 结合**：主 schema 四字段——step_by_step_analysis、reasoning_summary、relevant_pages、final_answer
- **SO 重解析器**：schema.model_validate 验证失败则重发 LLM，合规率达 100%（即使 8B 模型）
- **单次示例提示词（One-shot）**：每个 Prompt 加一对"问题→答案"示例

### 指令细化
- 解释自由度阈值问题（如"CEO"涵盖范围）需提前定义校准
- 手动构建验证集（比赛前一周问题生成器代码公开）

## 系统表现

- 速度：分批处理 25 问题，2 分钟完成全部 100 问
- 质量：Llama 3.3 70B 仅比 o3-mini 低几分，Llama 8B 也超过 80% 参赛者

## 结论

RAG 的魔力在于细节：高质量解析、高效检索、智能路由，以及 LLM 重排序和精心设计的提示词。
