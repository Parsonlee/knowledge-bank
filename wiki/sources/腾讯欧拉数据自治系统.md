# 腾讯欧拉如何打造数据自治系统

## 来源信息

- 作者：虎兴龙（腾讯数据工程专家，研发组长）
- 来源：DataFun 社区
- 标签：`AI-Agent/AI-BI`

## 核心要点

### 1. 核心理念：数据治理的耗散结构

**本质问题**：企业数据体系信息熵太大，数据治理即对抗熵增。

两种治理模式对比：
- **事后治理**（封闭系统对外做功）：先污染后治理，扫描元数据发现问题，被动修复
- **生产即治理**（耗散结构）[重点]：在数据生产和使用全程保持规范和自调节，主动防止熵增——类比人体，不做外部干预也能自调节维持健康

欧拉平台目标：建立从数仓到指标"生产即治理"的耗散结构，解决"治理难、维护难、使用难"三难。

### 2. 平台框架：三大子产品

1. **资产工场**：在湖/仓/流上构建标准化数仓模型
2. **指标中台 tMetric**：基于 Headless BI 理念，在数仓之上定义指标，提供统一指标服务
3. **数据发现**：面向消费端，将标准化生产的元信息集成为统一数据知识索引库

**评价体系**：资产分量化数据体系的信息熵（资产分越高，确定性越高），牵引治理平台落地。

### 3. 数仓建模：三层模型 + DataOps

**三层建模**：
- **概念模型**：定义业务域/主题域/业务过程，管理数据标准（词根/命名规范/度量单位），表名自动生成
- **逻辑模型**：星型/雪花模型，或可视化定义 pipeline，不绑定底层引擎
- **物理模型**：具体引擎（Hive/Spark/Clickhouse）实现

**典型数据问题（案例）**：
- 三个 ADS 表指标口径不一致（不同开发者，加工逻辑不对齐）
- 数据依赖错综复杂（同层/跨层/下层依赖上层）
- 过度冗余（DWD/ODS 存储占用大）
- 使用难（同名不同义、同义不同名，无分域分类）

**DataOps 理念**：[重点]
- 将数据工程软件工程化（CI/CD/CR + 版本/测试/质量）
- 面向表格的对象化开发：基础信息 + 生产代码 + Schema 定义 + 调度配置均可代码化
- Python + SQL 结合实现流程控制和代码复用

### 4. 指标中台 tMetric

**核心问题**：指标重复定义（同义不同名/同名不同义），口径不一致。

**Headless BI 方案**：[重点]
- 建立统一指标库，下游通过 SDK/API/类 SQL 查询指标
- "headless"即前后端分离：后端提供统一 API，上层多个可视化应用共用同一口径

**指标标准化定义**：
- 要素：度量（如视频总播放时长）+ 维度（性别/年龄）+ 业务限定（体育类）+ 统计周期（最近七天）
- 自动生成计算 SQL

**tMetric 体系架构**：
- 应用生态：决策智能/报表/实验/目标管理平台统一对接指标 API
- 查询层：缓存 + 查询协议 + 转换 + 路由策略
- 语义层：指标/维度标准化定义和元信息维护
- 数仓模型层：逻辑表/物理表定义 + 统一元数据服务
- 物化加速层：三种场景选择不同物化策略

**物化加速三种策略**：
1. **预计算 Cube**：常被观测、维度基数低、维度组合已知 → 查询快、成本低，灵活性低
2. **OLAP 引擎**（StarRocks/Clickhouse）：维度多、未来使用维度不确定 → 任意维度查询，按高频维度建物化视图
3. **MPP 内存计算**（Presto/Impala）：指标定义时快速预览验证 → 不是高频操作

### 5. 数据发现

- **统一元数据底座 Uni-Meta**：从各数据系统采集元数据，形成全域数据知识图谱
- 技术元数据（存储位置/结构/大小/格式）+ 业务元数据（业务含义/口径/分类）
- 上层数据发现产品：找数据 / 共享数据 / 申请数据 / 洞察数据

### 6. AI 在数据治理中的应用

- **AI 提升治理自动化**：自动补充表描述、自动分类数据（但需业务理解，AI 只能辅助）
- **AI for BI**（增强分析）：需要极高的数据治理水平做底座，事后治理无法支撑
- 核心洞察：**治理越好，AI 辅助效果越好**——规范化数据 + 标准化模板 → AI 能生成直接可运行的 SQL

> "治理难、维护难、使用难……数据治理的成果像沙子堆的碉堡，缺乏骨架，经不起风吹草动。" [重点]

> "如果数据未经治理，没有统一的数据标准和数据框架，那么即使把所有的元数据信息都采集，AI 的回答也会似是而非。" [高亮]

> "粗略的正确好过精确的错误。" [重点]
