\## 🚁 前言

上一篇我们主要介绍了 BM25 检索算法，作为 RAG 系统中检索部分的重要组成部分。但是，构建了 RAG 系统后，如何评估它的表现呢？是判断回答是否只是"看起来不错"，还是真正满足了用户的需求？这就需要一套科学的评估体系。

这次我们将学习**RAGAS 评估框架** ，它是专为评估 RAG 系统而设计的开源工具集。就像你不能只依靠感觉判断菜的好坏一样，RAG 系统的质量也需要客观的度量标准。RAGAS 提供了多个维度的评估指标，让我们能够准确判断 RAG 系统的性能。**虽然我们不需要每次评估系统都去使用RAGAS上所有的评价指标，但是有很多的指标设计是值得学习和参考的，相信看完它的指标设计，也能够给你许多在开发模型的时候带来启发的。**

\## 🐱 一、RAGAS 评估框架简介

\#\## 1.1 什么是 RAGAS

RAGAS 是一个专为检索增强生成(RAG)系统设计的评估框架，它提供了一套全面的指标，用于衡量 RAG 系统在不同维度的表现。与传统的文本生成评估方法不同，RAGAS 专注于 RAG 特有的挑战，例如信息检索的准确性、生成内容与检索上下文的一致性等。

\#\## 1.2 为什么需要 RAGAS

构建 RAG 系统时，我们通常会关注以下问题：

1.
   系统是否检索到了相关的信息？
2.
   检索的信息是否包含回答问题所需的全部内容？
3.
   生成的回答是否基于检索的上下文？
4.
   回答是否准确且相关？

传统的评估方法如 BLEU、ROUGE 等难以全面衡量这些方面。而 RAGAS 提供了更专业、更全面的评估指标，帮助开发者从多个角度评估和优化 RAG 系统。

\#\## 1.3 传统文本评估指标及其局限性

在深入了解 RAGAS 之前，我们先来了解一下传统的自然语言生成评估指标，以及它们为何不足以评估 RAG 系统：

\#\#\## BLEU (Bilingual Evaluation Understudy)

BLEU 是机器翻译和文本生成领域最流行的评估指标之一，它主要通过比较生成文本与参考文本之间的 n-gram 重叠度来衡量质量。

**计算原理** ：

*

*
  应用简短惩罚(Brevity Penalty)来避免过短输出获得不合理的高分
*
  最终分数是不同 n-gram 精确度的几何平均值

*

*

*

*
  对同义词和表达变化敏感

不过微调的时候这个指标还是很有用的，毕竟是一个很好的参考。

\#\#\## ROUGE (Recall-Oriented Understudy for Gisting Evaluation)

ROUGE 是一套用于评估自动摘要和机器翻译质量的指标。 它的主要作用是通过比较机器生成的文本（例如摘要或翻译）与人类创建的参考文本，来衡量生成文本的质量。 ROUGE 特别关注召回率（recall），也就是机器生成的文本中有多少内容与参考文本重叠或匹配。

**主要变体** ：

*
  **ROUGE-N** ：测量参考文本和生成文本之间 n-gram 的重叠度
*
  **ROUGE-L** ：基于最长公共子序列(LCS)的评估方法
*
  **ROUGE-S** ：考虑跳过二元组(Skip-bigram)的共现统计

*
  同样依赖表面文本匹配而非深层语义
*
  难以评估创新性表达
*
  对于 RAG 系统，不能评估检索质量和信息来源
*
  可能给冗长但低质量的输出高分

这些传统指标虽然在特定任务中有效，但在评估 RAG 系统时存在明显不足：它们无法衡量检索的质量、上下文利用情况、事实一致性和信息完整性。

\## 🦄 二、RAGAS 核心评估指标解析

RAGAS 框架提供了一套评估 RAG 系统（检索增强生成系统）的核心指标，就像给系统做全面体检的"检查单"。这些指标分别关注检索的信息全不全、准不准，以及生成的回答对不对、跑没跑题。下面解释每个指标的含义和作用，再配上理论公式和计算例子，让你既明白原理也能上手用。请尽量理解它评估的原理，因为理解它的原理收益是最大的，这部分我建议不是仅仅要求自己会用，而是理解原理。

\#\## 2.1 上下文召回率 (Context Recall) ------ "检索漏了多少关键信息？"

\#\#\## 通俗概念

想象你在图书馆找一本关于"恐龙"的书，找到了一本，但里面只讲了"恐龙长什么样"，没提"恐龙吃什么"。你会觉得信息不全，对吧？**上下文召回率** 就是检查 RAG 系统检索到的信息，有没有漏掉回答问题需要的关键内容。它衡量的是"检索能不能把重要点全找回来"。

\#\#\## 为什么重要

RAG 系统的第一步是检索。如果漏了关键信息，后面的生成再聪明也没用。比如你问"埃菲尔铁塔在哪里"，检索只说"它很高"，没提"巴黎"，回答肯定不靠谱。上下文召回率帮我们揪出这种"漏网之鱼"。

\#\## 2.4 答案相似度 (Answer Similarity) ------ "回答跟真相有多像？"

\#\#\## 通俗概念

你问"月亮离地球多远"，标准答案是"384,400公里"，系统回答"380,000公里"。数字挺接近，但不完全一样。**答案相似度** 检查生成的回答跟标准答案（Reference）有多吻合，分数越高说明回答越接近真相。它关注的是回答和真相在语义上的"亲密程度"。

\#\#\## 使用bert进行语义相似度计算

这里我们用bert给大家体验下如何使用向量模型计算两个句子之间的相似度。

    from transformers import BertTokenizer, BertModel
    import torch
    import numpy as np

    # 加载预训练的 BERT 模型和分词器
    tokenizer = BertTokenizer.from_pretrained('bert-base-uncased')
    model = BertModel.from_pretrained('bert-base-uncased')

    # 定义两个句子
    sentence1 = "我喜欢看电影。"
    sentence2 = "我很享受看电影。"

    # 对句子进行编码
    inputs1 = tokenizer(sentence1, return_tensors='pt', padding=True, truncation=True, max_length=128)
    inputs2 = tokenizer(sentence2, return_tensors='pt', padding=True, truncation=True, max_length=128)

    # 获取 BERT 输出
    with torch.no_grad():
            outputs1 = model(**inputs1)
            outputs2 = model(**inputs2)

    # 提取 [CLS] token 的表示（也可以用 mean pooling）
    cls_embedding1 = outputs1.last_hidden_state[:, 0, :].squeeze().numpy()
    cls_embedding2 = outputs2.last_hidden_state[:, 0, :].squeeze().numpy()

    # 计算余弦相似度
    cos_sim = np.dot(cls_embedding1, cls_embedding2) / (np.linalg.norm(cls_embedding1) * np.linalg.norm(cls_embedding2))
    print(f"两个句子的余弦相似度: {cos_sim:.4f}")

输出的结果是：

    两个句子的余弦相似度: 0.9979

\#\## 2.5 回答相关性 (Answer Relevance) ------ "回答跑题了吗？"

\#\#\## 通俗概念

你问"苹果多少钱一斤"，系统回答"苹果公司很赚钱"，完全跑题。**回答相关性** 检查回答跟问题有多搭，跑题的得分低。它不需要标准答案，而是看回答能不能对齐问题。

\#\#\## 为什么重要

用户要的是解决问题，回答不相关再正确也没用。回答相关性确保回答紧扣主题，尤其在没有标准答案时特别实用。

\#\#\## 理论公式

1.
   用大语言模型（LLM）根据回答反推出可能的问题（Generated Questions）。
2.
   用嵌入模型提取实际问题（Question）和生成问题的语义向量。
3.
   计算实际问题与生成问题的平均相似度：

*
  ：生成的问题数量。
*
  ：实际问题。
*
  ：第 个生成的问题。
*
  ：余弦相似度。

\#\## 3.5 根据评估结果优化 RAG 系统

现在，让我们看看如何使用 RAGAS 对 RAG 系统进行综合评估，并根据评估结果进行优化。

**分析指标表现** ：

*
  如果上下文召回率低，考虑增加检索结果数量或使用更先进的检索算法
*
  如果上下文精确度低，可能需要优化检索排序或过滤无关文档
*
  如果答案正确性低，可能需要改进提示词或微调生成模型
*
  如果响应相关性低，检查提示词是否引导模型聚焦于问题
